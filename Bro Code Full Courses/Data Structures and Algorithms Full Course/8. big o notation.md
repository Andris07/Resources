# Big O Notation

# What is Big O Notation?

**Big O Notation** is a mathematical way of describing how the performance of an algorithm changes as the amount of data increases.

It measures the **time complexity** (how long an algorithm takes to execute) and **space complexity** (how much memory an algorithm uses).

Rather than measuring the exact execution time, Big O describes how an algorithm **scales** as the input size grows.

---

# What is **n**?

In Big O notation, **n** represents the **size of the input**.

For example:

* The number of elements in an array.
* The number of nodes in a tree.
* The number of vertices in a graph.
* The number of items in a list.

Example:

```text
Array:

[5, 8, 12, 15, 27]

n = 5
```

As **n** increases, algorithms may become faster or slower depending on their complexity.

---

# Time Complexity

**Time Complexity** describes how the execution time grows as the input size (**n**) increases.

Example:

```java
for (int i = 0; i < n; i++) {
    System.out.println(i);
}
```

The loop executes **n** times.

Time Complexity:

```text
O(n)
```

---

# Space Complexity

**Space Complexity** describes how much additional memory an algorithm requires while it runs.

Example:

```java
int[] numbers = new int[n];
```

The array stores **n** integers.

Space Complexity:

```text
O(n)
```

Example:

```java
int sum = 0;
```

Only one variable is used regardless of the input size.

Space Complexity:

```text
O(1)
```

---

# Common Big O Complexities

The following complexities are ordered from **fastest** to **slowest**.

| Complexity | Performance                     |
| ---------- | ------------------------------- |
| O(1)       | ⭐ Excellent                     |
| O(log n)   | ⭐ Very Fast                     |
| O(n)       | ✅ Good                          |
| O(n log n) | ✅ Efficient                     |
| O(n²)      | ❌ Slow                          |
| O(n³)      | ❌ Very Slow                     |
| O(2ⁿ)      | 🚨 Extremely Slow               |
| O(n!)      | 💀 Impractical for large inputs |

---

# O(1) — Constant Time

The execution time never changes, regardless of the input size.

Example:

```java
array[5];
```

Accessing an array element by its index always takes the same amount of time.

Other examples:

* Array index access
* Stack `push()`
* Stack `pop()`
* Queue `offer()`
* Queue `poll()`
* LinkedList `addFirst()`
* LinkedList `removeFirst()`

---

# O(log n) — Logarithmic Time

The amount of work grows very slowly as the input size increases.

Each step removes about half of the remaining data.

Example:

```text
1024 elements

↓

512

↓

256

↓

128

↓

...
```

Example algorithm:

* Binary Search

---

# O(n) — Linear Time

The algorithm processes every element once.

Example:

```java
for (int value : array) {
    System.out.println(value);
}
```

Examples:

* Linear Search
* Finding the maximum value
* Copying an array

---

# O(n log n) — Linearithmic Time

A very efficient complexity for sorting algorithms.

Examples:

* Merge Sort
* Heap Sort
* Average-case Quick Sort

---

# O(n²) — Quadratic Time

Every element is compared with every other element.

Example:

```java
for (...) {
    for (...) {

    }
}
```

Examples:

* Bubble Sort
* Selection Sort
* Insertion Sort (worst case)

---

# O(n³) — Cubic Time

Three nested loops.

Example:

```java
for (...) {
    for (...) {
        for (...) {

        }
    }
}
```

Examples:

* Some matrix multiplication algorithms
* Brute-force graph algorithms

---

# O(2ⁿ) — Exponential Time

Each additional element roughly doubles the amount of work.

Examples:

* Recursive Fibonacci (naive implementation)
* Brute-force subset generation

---

# O(n!) — Factorial Time

The slowest common complexity.

Examples:

* Brute-force Traveling Salesman Problem
* Generating every possible permutation

# Time Complexity vs Space Complexity

Sometimes an algorithm is made faster by using more memory.

Example:

* **HashMap**

  * Fast lookups: **O(1)** average time.
  * Requires additional memory: **O(n)** space.

Sometimes an algorithm uses very little memory but takes longer to execute.

Finding the right balance between **Time Complexity** and **Space Complexity** is one of the most important goals in algorithm design.

---

# Why Is Big O Important?

Big O Notation allows developers to compare algorithms independently of hardware or programming language.

Understanding Big O helps developers:

* Choose the most efficient algorithm.
* Predict how software will scale.
* Optimize performance.
* Reduce execution time.
* Reduce memory usage.

Big O is one of the most fundamental concepts in Data Structures and Algorithms because it provides a standard way to measure and compare algorithm efficiency.
